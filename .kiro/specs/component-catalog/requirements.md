# Requirements Document

## Introduction

This specification defines requirements for the **Component Catalog** — a Storybook-based browsing and documentation layer for React components and page templates generated by the DSL pipeline. The catalog provides visual previews of all component variants, displays associated DSL source code and compiled JSON, and optionally renders DSL-based previews alongside React previews for side-by-side comparison.

**Context**: The project already has a `preview/` Vite app with 25+ React components and 4 showcase pages, plus `examples/` containing `.dsl.ts` files that compile to JSON and render to PNG. Today there is no unified browsing experience — developers must run the dev server and navigate manually, or use CLI commands to render individual DSL files. A Storybook-powered catalog centralizes discovery, variant exploration, and DSL artifact inspection.

**Target users**:
- Component developers iterating on React components and their DSL counterparts
- Designers reviewing component variants and layout fidelity
- AI Skills that generate components and need a catalog registry for discovery

## Requirements

### Requirement 1: Storybook Integration

**Objective:** As a component developer, I want to browse all React components in a Storybook catalog, so that I can discover, preview, and interact with components without navigating raw source files.

#### Acceptance Criteria

1. The Catalog shall use Storybook as the catalog framework, configured for the existing `preview/` Vite + React 19 + TypeScript project.
2. The Catalog shall automatically discover and display all React components located in `preview/src/components/`.
3. When a component is selected in the sidebar, the Catalog shall render a live, interactive preview of that component.
4. The Catalog shall organize components hierarchically in the sidebar (e.g., grouped by category or alphabetically).

### Requirement 2: Component Variant Stories

**Objective:** As a component developer, I want each component's story to show all its visual variants, so that I can verify every state renders correctly without manually toggling props.

#### Acceptance Criteria

1. The Catalog shall include stories that render all meaningful variants for each component (e.g., Button: primary, secondary, outline, disabled, sizes).
2. When a component accepts props that produce visual variants, the Catalog shall provide Storybook controls (args/knobs) to interactively adjust those props.
3. The Catalog shall include a "All Variants" story per component that displays every variant simultaneously in a grid or list layout.

### Requirement 3: Page Template Catalog

**Objective:** As a developer or designer, I want to browse page templates (multi-section compositions) in the catalog, so that I can preview and reuse full-page layouts.

#### Acceptance Criteria

1. The Catalog shall include a dedicated "Pages" section in the sidebar for page template stories.
2. The Catalog shall display page templates from `preview/src/pages/` as full-page preview stories.
3. When a page template is selected, the Catalog shall render it at full width to reflect realistic viewport layout.
4. The Catalog shall support viewport switching (desktop, tablet, mobile) for page template stories.

### Requirement 4: DSL Source Code Display

**Objective:** As a component developer, I want to view the DSL source code (`.dsl.ts`) associated with a component alongside its React preview, so that I can understand and iterate on the DSL definition.

#### Acceptance Criteria

1. Where a component has a corresponding `.dsl.ts` file in `examples/`, the Catalog shall display the DSL source code in a syntax-highlighted code panel.
2. When the DSL source code panel is shown, the Catalog shall provide a copy-to-clipboard button.
3. If a component does not have a corresponding `.dsl.ts` file, the Catalog shall show a placeholder indicating no DSL definition exists yet.

### Requirement 5: Compiled JSON Display

**Objective:** As a component developer, I want to inspect the compiled JSON output from a DSL definition, so that I can verify the intermediate representation before export.

#### Acceptance Criteria

1. Where a component has a corresponding `.dsl.ts` file, the Catalog shall display the compiled JSON output in a collapsible, syntax-highlighted panel.
2. The Catalog shall generate the JSON by compiling the `.dsl.ts` file through the existing `@figma-dsl/compiler` pipeline.
3. When the JSON panel is expanded, the Catalog shall render the JSON with proper indentation and syntax highlighting.
4. The Catalog shall provide a copy-to-clipboard button for the compiled JSON.

### Requirement 6: DSL-Rendered Preview

**Objective:** As a component developer, I want to see a DSL-rendered PNG preview alongside the React preview, so that I can compare rendering fidelity between the two approaches.

#### Acceptance Criteria

1. Where a component has a corresponding `.dsl.ts` file, the Catalog shall display a DSL-rendered PNG image alongside the React preview.
2. The Catalog shall generate the PNG by running the `.dsl.ts` file through the existing `@figma-dsl/renderer` pipeline.
3. When both React and DSL previews are available, the Catalog shall display them side-by-side with labels ("React" / "DSL").
4. The Catalog shall provide a toggle or tab to switch between "React only", "DSL only", and "Side-by-side" views.
5. If DSL rendering fails, the Catalog shall display an error message with the failure reason instead of a broken image.

### Requirement 7: Component-DSL File Association

**Objective:** As a developer, I want the catalog to automatically associate React components with their DSL files, so that I do not need to manually configure mappings.

#### Acceptance Criteria

1. The Catalog shall use a naming convention to associate React components with DSL files (e.g., `Button` component → `button.dsl.ts` in `examples/`).
2. Where automatic association fails, the Catalog shall support an optional manual mapping configuration.
3. When a new `.dsl.ts` file is added to `examples/`, the Catalog shall pick it up without requiring code changes to the catalog itself.
