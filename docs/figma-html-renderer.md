## 1. Project Overview

**Confidence**: 0.96 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

Figma HTML Renderer is a Python CLI tool that converts Figma `.fig` files into interactive HTML pages — no Figma app or API required. It implements a strict five-stage sequential pipeline:

1. **Parse** — Extract and decode the binary `.fig` format
2. **Build Tree** — Reconstruct the node hierarchy from flat data
3. **Classify** — Identify interactive (clickable) elements via heuristic rules
4. **Render** — Rasterize the design to PNG via PyCairo
5. **Output** — Generate HTML with the PNG as background and transparent interactive overlays

The architecture is deliberately simple — no plugin system, no abstract base classes, no dependency injection. Each stage has a single class, a dedicated dataclass for inter-stage data, and a corresponding test module.

## 4. Pipeline Architecture

**Confidence**: 0.96 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

```
.fig file (Path)
    │
    ▼
[Stage 1: FigParser.parse()]
    │  Output: ParseResult
    │    .node_changes: list[dict]     — flat list of all Figma nodes
    │    .images_dir: Path             — extracted image assets
    │    .fig_zip: ZipFile             — open handle for resource access
    │    .meta: dict | None            — parsed meta.json
    ▼
[Stage 2: TreeBuilder.build_tree()]
    │  Output: TreeResult
    │    .root: dict                   — DOCUMENT node (guid=(0,0))
    │    .guid_map: dict               — all nodes indexed by GUID tuple
    │    .design_pages: list[dict]     — filtered CANVAS nodes
    ▼
[Page Selection: _select_page()]
    ▼
[Stage 3: NodeClassifier.classify()]
    │  Output: ClassifyResult
    │    .canvas_root: dict            — deep copy of full tree (for PNG)
    │    .interactive_nodes: list[dict] — clickable elements
    ▼
[Stage 3.5: SectionDetector.detect()]
    │  Output: list[Section]
    │    .name, .y_start, .y_end, .index
    ▼
[Stage 4: CanvasRenderer.render()]  (once per section)
    │  Output: RenderResult
    │    .png_path, .width, .height, .offset_x, .offset_y
    ▼
[Stage 5: HTMLOutputGenerator.generate() or .generate_combined()]
    │  Output: Path to HTML file
    ▼
HTML file(s) on disk
```

### Key Architectural Decisions

1. **Immutable inter-stage contracts** — Each stage produces a `@dataclass`. Stages never mutate upstream results.
2. **Context manager for resource lifecycle** — `FigParser` manages temp directory. Entire pipeline runs inside `with FigParser() as parser:`.
3. **Dictionary-centric data model** — Nodes remain as `dict[str, Any]` throughout (no domain model classes).
4. **Interactive nodes in canvas** — The classifier deep-copies the full tree. PNG is a complete visual representation; overlays are transparent on top. (This deviates from the original plan which specified excluding interactive nodes from rendering.)
5. **Single vs. multi-section branching** — CLI branches at section detection: single section → one render + one generate; multiple sections → loop render + combined generate.

> `cli.py:53-134`, all stage modules

## 6. Pipeline Stage API

**Confidence**: 0.95 | **Consensus**: Full | **Sources**: All agents

### Stage 1: FigParser

```python
class FigParser:
    def __enter__(self) -> Self: ...
    def __exit__(self, ...): ...
    def parse(self, fig_path: Path) -> ParseResult: ...
```

**Must be used as a context manager** — `parse()` outside `with` raises `RuntimeError`.

**Decoding strategy**: Primary `figformat.decodefig.decode()`, fallback `figformat.kiwi.decode()` when ImportError contains "converter.positioning".

**Validation**: Checks ZIP magic bytes `PK`. Raises `FileNotFoundError` for missing files, `ValueError` for non-ZIP.

### Stage 2: TreeBuilder

```python
class TreeBuilder:
    def build_tree(self, node_changes: list[dict]) -> TreeResult: ...
```

**GUID handling**: Lists converted to tuples. Root is `(0, 0)`. Children sorted by `parentIndex.position`.

**Filtering**: CANVAS nodes named `^internal\b` or `^---$` are excluded.

**Raises** `ValueError` for missing root or no design pages.

### Stage 3: NodeClassifier

```python
class NodeClassifier:
    def __init__(self, extra_rules: list[ClassificationRule] | None = None): ...
    def classify(self, root: dict, guid_map: dict) -> ClassifyResult: ...
```

**4 default rules** (see [Classification System](#7-classification-system)).

**Extension**: `NodeClassifier(extra_rules=[my_rule])`. Rule signature: `(node: dict, ctx: dict) -> bool`.

**Behavior**: Children of interactive nodes are not recursed into.

### Stage 4: CanvasRenderer

```python
class CanvasRenderer:
    def render(self, canvas_root: dict, images_dir: Path, output_path: Path,
               section_bounds: tuple[float, float] | None = None) -> RenderResult: ...
```

**Supported types**: FRAME, COMPONENT, COMPONENT_SET, INSTANCE, ROUNDED_RECTANGLE, RECTANGLE, TEXT, ELLIPSE, VECTOR (fallback to rect).

**Fills**: SOLID (RGBA) and IMAGE (SHA1 hash lookup, Pillow fallback).

**Text**: Baseline-accurate path via `derivedTextData.baselines`, or fallback with `fontSize * 1.2` line spacing.

### Stage 4.5: SectionDetector

```python
class SectionDetector:
    def __init__(self, height_threshold: float = 2000.0): ...
    def detect(self, page_frame: dict, mode: str = "auto",
               manual_breakpoints: list[float] | None = None) -> list[Section]: ...
```

### Stage 5: HTMLOutputGenerator

```python
class HTMLOutputGenerator:
    def generate(self, render_result, interactive_nodes, guid_map, output_path,
                 options=None, *, offset_x=0.0, offset_y=0.0,
                 section_bounds=None) -> Path: ...
    def generate_combined(self, section_render_results, section_bounds_list,
                          interactive_nodes, guid_map, output_path,
                          options=None, *, offset_x=0.0, offset_y=0.0) -> Path: ...
    def generate_index(self, section_html_paths, section_names, output_path) -> Path: ...
```

## 7. Classification System

**Confidence**: 0.95 | **Consensus**: Full | **Sources**: All agents

### Default Rules

| Rule | Priority | Criteria |
|------|----------|----------|
| `rule_hyperlink` | 1st | `node.hyperlink is not None` |
| `rule_cta_button` | 2nd | FRAME + `stackMode` truthy + `cornerRadius > 20` |
| `rule_nav_link` | 3rd | TEXT + Y position < 150px + `fontSize <= 16` |
| `rule_cta_keyword` | 4th | TEXT + name/content matches CTA keywords |

**CTA keywords**: `book`, `shop`, `buy`, `subscribe`, `join`, `sign`, `follow`, `contact` (regex with word boundaries).

### Custom Rules

```python
def my_rule(node: dict, ctx: dict) -> bool:
    """ctx contains page_width and page_height."""
    return node.get("name") == "ClickMe"

classifier = NodeClassifier(extra_rules=[my_rule])
```

Custom rules are appended after the 4 defaults.

### Rule Caveats

- **`rule_cta_button`**: Corner radius threshold of 20 is hardcoded. May miss material-design buttons with lower radii.
- **`rule_nav_link`**: Uses node's local Y transform, not absolute position. Deeply nested text may be misclassified.
- **`rule_cta_keyword`**: CTA keywords list is not configurable without source modification.

> `classifier.py:1-150`

## 8. Section Detection

**Confidence**: 0.93 | **Consensus**: Full | **Sources**: All agents

Addresses the problem of very tall pages producing enormous single PNGs.

### Modes

| Mode | Behavior |
|------|----------|
| `auto` (default) | Only activates above `height_threshold` (2000px). Finds full-width (≥95% page width) "anchor" nodes, splits at midpoints between adjacent anchors. |
| `none` | Returns single full-page section regardless of height. |
| `manual:Y1,Y2,...` | Splits at user-specified Y coordinates. |

### Section Naming

1. Search anchor children (2 levels deep) for TEXT nodes with `fontSize >= 24`
2. Take first 40 characters, lowercase, spaces → hyphens
3. Fallback to anchor node name, then `section-{index}`

### Output

Multi-section produces:
```
output/
    {page_name}_section-0.png
    {page_name}_section-1.png
    {page_name}.html  # Single combined HTML with stacked sections
```

> `section_detector.py:1-205`, `cli.py:82-134`

## 9. HTML Output Format

**Confidence**: 0.95 | **Consensus**: Full | **Sources**: All agents

### Structure

The generated HTML has three layers:

1. **CSS**: Reset styles, centered flexbox, `.canvas-container` with PNG background, `.interactive-overlay` with absolute positioning and hover transitions (`rgba(59, 130, 246, 0.1)` on hover)
2. **Overlays**: `<a>` for hyperlinked nodes (with `target="_blank"`), `<div>` for others. Positioned via parent-chain translation accumulation.
3. **JavaScript**: Click handlers dispatch `figma-click` CustomEvent with `{ name, type, timestamp }`, bubbling up from overlay elements.

### Output Modes

| Mode | Description |
|------|-------------|
| Linked PNG (default) | HTML references external PNG via relative path |
| Embedded (`--embed`) | PNG base64-encoded into `data:` URI |
| Debug (`--debug`) | Red border outlines on overlays |

### Browser Integration

```javascript
document.addEventListener('figma-click', (e) => {
    console.log('Clicked:', e.detail.name, e.detail.type);
});
```

## 15. Design Patterns & Architecture Decisions

**Confidence**: 0.92 | **Consensus**: Majority | **Sources**: Architect, implicit others

1. **Pipes and Filters** — Each stage produces a `@dataclass` result. No stage calls another directly; `cli.py` is the sole composition point.
2. **Context Manager** — `FigParser` manages temp directory lifecycle. Ensures cleanup even on exceptions.
3. **Dictionary-centric Data Model** — Nodes remain as `dict[str, Any]` throughout. Avoids mapping layer overhead, at the cost of no type safety on property access.
4. **Rule-based Classification** — Strategy pattern with open extension via `extra_rules`. Each rule is a simple function.
5. **Transform Stack Rendering** — Cairo's `save()`/`restore()` + `transform()` for correct nested affine transform composition.
6. **No Abstraction Layers** — No interfaces, abstract base classes, or dependency injection. Direct imports, concrete instantiation.
7. **`string.Template` for HTML** — No Jinja2 or Mako. Zero template-layer dependencies.
8. **Plan Deviation** — Interactive nodes are kept in the canvas (rendered in PNG), contrary to the original plan which specified exclusion. This simplifies rendering but means the PNG is always a complete visual.

### Module Dependency Graph

```
cli.py → parser, tree_builder, classifier, section_detector, renderer, output
output.py → renderer (RenderResult type only)
__main__.py → cli
preview_server.py → (stdlib only)
```

Acyclic, shallow. Only `parser.py` and `renderer.py` have external package dependencies.

## 16. Security Considerations

**Confidence**: 0.88 | **Consensus**: Validated (3/3 validators confirmed all claims)

### XSS via Node Names

Node names from `.fig` files are inserted into HTML `data-element` attributes via f-strings **without HTML escaping**. A malicious `.fig` file with a node named `foo" onclick="alert(1)` could inject arbitrary HTML attributes. No HTML escape functions exist anywhere in the source.

> `output.py:145-170, 290-316`

### XSS via Hyperlink URLs

Hyperlink URLs are inserted into `href` attributes **without URL scheme validation**. `javascript:` URLs from `.fig` files would execute JavaScript when clicked.

> `output.py:149-153`

### ZIP Path Traversal

`ZipFile.extractall()` is called without member path validation. Python 3.12+ has built-in protections, but the project supports Python 3.10+ (`requires-python = ">=3.10"`), leaving 3.10-3.11 potentially vulnerable to zip-slip attacks.

> `parser.py:87`

### Preview Server

- Logging suppressed (`log_message` is no-op) — harder to detect issues
- `Access-Control-Allow-Origin: *` on reload check endpoint
- Intended for local development only

### Denial of Service

No limits on input file size, page dimensions, or node count. A crafted `.fig` file with extreme dimensions could attempt to allocate gigabytes of memory for the Cairo surface.

## 17. Performance Characteristics

**Confidence**: 0.85 | **Consensus**: Validated (3/3 validators confirmed)

### Memory Usage Hotspots

1. **ZIP extraction** (`parser.py:87`): All files extracted to disk. Large `.fig` files with many images consume significant temp disk space.
2. **Deep copy of tree** (`classifier.py:123`): `copy.deepcopy(root)` duplicates the entire node tree. For documents with thousands of nodes, this doubles tree memory usage.
3. **Cairo surface allocation** (`renderer.py:83`): `cairo.ImageSurface(FORMAT_ARGB32, width, height)` allocates `4 * width * height` bytes. A 1440x50000 page ≈ 274 MB. Section splitting mitigates this.
4. **Base64 encoding** (`output.py:117-119`): When `embed_png=True`, entire PNG read + base64-encoded (+33% size increase).

### CPU Complexity

- Tree building: O(n) indexing + O(n log n) child sorting
- Classification: O(n * r) where r = number of rules (4 default)
- Rendering: O(n) traversal with per-node Cairo operations
- Overlay positioning: O(depth) parent-chain walk per interactive node

No parallelism — the pipeline is entirely single-threaded.

## 18. Known Gaps & Limitations

**Confidence**: 0.90 | **Consensus**: Full/Validated | **Sources**: All agents + validators
