## Overview
**Confidence**: 0.96 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

The `@figma-dsl/capturer` package is a lightweight, specialized browser automation library that captures screenshots of React components and web pages using Playwright's headless Chromium engine. It provides two primary async functions — `capture()` for rendering React components in an isolated HTML context, and `captureUrl()` for screenshotting live URLs — and is consumed exclusively by the `@figma-dsl/cli` package within the figma-dsl monorepo.

The package serves the visual regression testing and component preview pipeline: it renders components to PNG images that can be compared against DSL-rendered reference images.

## Public API
**Confidence**: 0.97 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

### `capture(componentPath, props, outputPath, options): Promise<CaptureResult>`

Renders a React component to a PNG screenshot.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `componentPath` | `string` | Yes | ES module path to the React component (default export) |
| `props` | `Record<string, unknown>` | Yes | Props to pass to the component (JSON-serialized) |
| `outputPath` | `string` | Yes | File system path for the output PNG |
| `options` | `CaptureOptions` | Yes | Viewport, selector, background, and scale settings |

**Behavior**:
1. Launches headless Chromium via Playwright
2. Generates a minimal HTML page with an inline ES module script
3. Dynamically imports the component and renders it with `createRoot` + `createElement`
4. Waits for `networkidle`, then waits for the CSS selector (default: `#root > *`)
5. Takes an element-level PNG screenshot and extracts bounding box dimensions
6. Writes the PNG to disk (creating parent directories as needed)
7. Returns `CaptureResult` with path and dimensions

**Evidence**: `src/capturer.ts:18-74`

### `captureUrl(url, outputPath, options): Promise<CaptureResult>`

Navigates to a live URL and captures a screenshot.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `url` | `string` | Yes | URL to navigate to |
| `outputPath` | `string` | Yes | File system path for the output PNG |
| `options` | `CaptureOptions` | Yes | Viewport, selector, background, and scale settings |

**Behavior**: Identical to `capture()` except it uses `page.goto(url)` instead of generating HTML with an inline React component. Same wait strategy, screenshot logic, and cleanup.

**Evidence**: `src/capturer.ts:76-114`

## Type Interfaces
**Confidence**: 0.96 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

### CaptureOptions

```typescript
interface CaptureOptions {
  viewport: { width: number; height: number };  // Required: browser viewport size
  selector?: string;                            // CSS selector to screenshot (default: '#root > *')
  background?: 'white' | 'transparent';         // Page background (default: 'white')
  deviceScaleFactor?: number;                   // Pixel density (default: 1)
}
```

- `viewport` is required and sets the browser context dimensions
- `selector` defaults to `#root > *` — the first child of the React root div
- `background` is applied as a CSS style on the HTML body; only two values supported
- `deviceScaleFactor` maps directly to Playwright's context option

**Evidence**: `src/capturer.ts:5-10`

### CaptureResult

```typescript
interface CaptureResult {
  pngPath: string;   // Absolute path to the saved PNG
  width: number;     // Element width (or viewport width as fallback)
  height: number;    // Element height (or viewport height as fallback)
}
```

If `element.boundingBox()` returns null, dimensions fall back to the viewport size. Callers cannot distinguish between "element matched viewport size" and "bounding box unavailable."

**Evidence**: `src/capturer.ts:12-16`, `src/capturer.ts:66-70`

## Browser Lifecycle & Data Flow
**Confidence**: 0.97 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

Both functions follow an identical lifecycle pattern:

```
chromium.launch({ headless: true })
  → browser.newContext({ viewport, deviceScaleFactor })
    → context.newPage()
      → [page.setContent() or page.goto()] with waitUntil: 'networkidle'
      → page.waitForSelector(selector, { timeout: 5000 })
      → element.screenshot({ type: 'png' })
      → element.boundingBox()
      → mkdirSync + writeFileSync
    → context.close()
  → [finally] browser.close()
```

Key design decisions:
- **Ephemeral browsers**: Each function call launches and destroys a full Chromium process — no connection pooling or reuse
- **try-finally cleanup**: `browser` is declared outside the try block; the finally block guarantees `browser.close()` even on errors
- **Context before browser**: `context.close()` is called inside the try block before returning; `browser.close()` in the finally block handles any remaining cleanup

**Evidence**: `src/capturer.ts:24-31,64,71-73` (capture), `src/capturer.ts:81-88,104,111-113` (captureUrl)

## HTML Generation & React Rendering
**Confidence**: 0.88 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

The `capture()` function generates a minimal HTML document with an inline ES module script:

1. Creates a `<div id="root">` container
2. Sets body background via CSS based on `options.background`
3. Inline `<script type="module">` that:
   - Imports the component from `componentPath` as the default export
   - Imports `createRoot` from `react-dom/client`
   - Calls `React.createElement(Component, props)` with JSON-deserialized props
   - Renders into the root div

**Critical assumptions**:
- `componentPath` must be a resolvable ES module URL
- Component must be the **default export**
- React 18+ is required (`createRoot` API from `react-dom/client`)
- Props are serialized via `JSON.stringify` — functions, symbols, and circular references will fail or be silently dropped

**Evidence**: `src/capturer.ts:33-48`

## Wait Strategies
**Confidence**: 0.90 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

Two sequential wait mechanisms are used:

1. **Network idle**: `waitUntil: 'networkidle'` — waits until network activity stops for 500ms (Playwright default). Applied to both `page.setContent()` and `page.goto()`.

2. **Selector presence**: `page.waitForSelector(selector, { timeout: 5000 })` — waits up to 5 seconds for the CSS selector to match a DOM element.

**Caveats**:
- The 5-second selector timeout is **hardcoded** and not configurable
- `networkidle` may never trigger if the page has persistent connections (WebSocket, long-polling)
- No visual stability check — screenshots may capture mid-animation or before final layout
- If the selector doesn't match within 5 seconds, a descriptive error is thrown: `"Element not found: ${selector}"`

**Evidence**: `src/capturer.ts:50-56,90-96`

## Error Handling & Failure Modes
**Confidence**: 0.91 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

### Errors that propagate to the caller

| Error | Cause | Behavior |
|-------|-------|----------|
| Browser launch failure | Chromium not installed / system resources | Playwright throws |
| Selector timeout | Element not found in 5s | Custom error: `"Element not found: ${selector}"` |
| Navigation failure | Invalid URL or network error | Playwright throws |
| File write failure | Permissions, disk full | Node.js `writeFileSync` throws |
| JSON serialization failure | Circular references in props | `JSON.stringify` throws TypeError |
| Screenshot failure | Element detached during capture | Playwright throws |

### Errors that are silently swallowed

- **Module import errors** inside the generated HTML page (Playwright doesn't propagate inline script errors)
- **React render errors** (caught by page error handler, not surfaced to caller)
- **Console errors/warnings** during page setup

### Cleanup guarantee

The `finally` block ensures `browser.close()` is always called. No retry logic — a single failure aborts the entire operation.

**Evidence**: `src/capturer.ts:54-56,71-73` (capture), `src/capturer.ts:94-96,111-113` (captureUrl)

## File System Operations
**Confidence**: 0.92 | **Consensus**: Full | **Sources**: Architect, Analyst

After capturing the screenshot buffer:

```typescript
mkdirSync(dirname(outputPath), { recursive: true });
writeFileSync(outputPath, screenshot);
```

- `recursive: true` creates parent directories if they don't exist
- Operations are **synchronous** (block execution until complete)
- No validation of `outputPath` format (no `.png` extension check)
- No path sanitization — caller is responsible for safe paths

## Security Considerations
**Confidence**: 0.79 | **Consensus**: Single Source (verified) | **Sources**: Analyst

The package has no runtime input validation. Security relies entirely on callers providing safe inputs.

| Vector | Risk | Mitigation |
|--------|------|------------|
| Module path injection | `componentPath` used directly in `import()` | Caller must validate |
| Path traversal | `outputPath` written without sanitization | Caller must validate |
| URL injection | `captureUrl()` accepts any URL including internal addresses | Caller must validate |
| Props serialization | Large/circular objects can DOS via `JSON.stringify` | None |
| Viewport DOS | No upper bound on viewport dimensions | None |

Built-in mitigations: headless mode only, `deviceScaleFactor` defaults to 1, 5-second selector timeout.

**Evidence**: `src/capturer.ts:18-23,34,61-62,76-79`

## Known Limitations
**Confidence**: 0.88 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

1. **React-only**: `capture()` hardcodes React 18+ rendering. No support for other frameworks.
2. **Default export only**: Component must be the default export of the ES module.
3. **Fixed selector timeout**: 5-second timeout is hardcoded — not configurable.
4. **Limited background options**: Only `'white'` or `'transparent'`.
5. **No browser reuse**: Each call launches a fresh Chromium process — inefficient for batch operations.
6. **No test coverage**: Package has zero test files. Tested indirectly via CLI integration tests.
7. **Swallowed page errors**: Module import failures and React render errors inside the generated page are not propagated to the caller.
8. **Ambiguous dimension fallback**: When `boundingBox()` returns null, viewport dimensions are returned silently — callers cannot detect this case.
9. **Unbounded networkidle**: Pages with persistent connections may cause the function to hang indefinitely.
