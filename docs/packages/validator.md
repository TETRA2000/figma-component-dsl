## Overview
**Confidence**: 0.96 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

The `@figma-dsl/validator` package is a rule-based validation system that enforces component structure and code patterns for DSL compatibility. It validates React components against 10 structural, styling, and type-safety rules, reporting errors (blocking violations) and warnings (best-practice recommendations).

The validator uses a plugin-style rule architecture with regex-based pattern matching (no AST parsing), keeping the package lightweight with zero external dependencies beyond `@figma-dsl/core`. It integrates into the CLI as the `figma-dsl validate` command and is also available as a programmatic API.

## Public API
**Confidence**: 0.98 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

### `validateComponent(componentDir: string, options?: ValidatorOptions): Promise<ValidationResult>`

Validates a single component directory. Expects directory structure matching component name (e.g., `Button/Button.tsx`). Builds context from filesystem, selects rules, executes sequentially, and returns partitioned results.

### `validateAll(baseDir: string, options?: ValidatorOptions): Promise<Map<string, ValidationResult>>`

Batch validates all component directories in a base path. Auto-discovers components by checking for `{dirName}/{dirName}.tsx` pattern. Returns `Map<string, ValidationResult>` keyed by component name. Returns empty map if baseDir doesn't exist.

### `allRules`

Array of all 10 `ValidationRule` objects. Exported for inspection or custom filtering.

### Exported Types

`ValidationError`, `ValidationResult`, `ValidatorOptions`, `ValidationRule`, `ValidationContext`

**Evidence**: `src/index.ts`, `src/validator.ts:75-126`

## Context Building
**Confidence**: 0.97 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

The `buildContext()` function (validator.ts lines 12-59) constructs a `ValidationContext` from the filesystem:

1. Derives `componentName` from directory basename
2. Checks for and reads (via `existsSync` + `readFileSync`):
   - `{ComponentName}.tsx` — React component source
   - `{ComponentName}.module.css` — CSS Module stylesheet
   - `{ComponentName}.figma.tsx` — Figma Code Connect binding
   - `../index.ts` — Parent barrel export
3. Resolves tokens path:
   - Uses explicit `options.tokensPath` if provided
   - Falls back to auto-detection: `../tokens.css`
   - Missing tokens.css is not an error; rules handle gracefully

**Component directory convention**:
```
components/
├── Button/
│   ├── Button.tsx              # Required for detection
│   ├── Button.module.css       # Expected
│   └── Button.figma.tsx        # Expected
├── Card/
│   ├── Card.tsx
│   ├── Card.module.css
│   └── Card.figma.tsx
└── index.ts                    # Parent barrel export
```

## Validation Rules Catalog
**Confidence**: 0.98 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

### Error Rules (6) — Block validation

| Rule ID | Purpose | Detection Method |
|---------|---------|-----------------|
| `css-modules` | Requires CSS Module import in `.tsx` | Regex: `/import\s+\w+\s+from\s+['"]\.\/\w+\.module\.css['"]/` |
| `no-inline-style` | Forbids inline `style={{}}` attributes | Per-line regex: `/style=\{\{/` with line numbers |
| `classname-prop` | Component must accept `className` prop and compose classnames | Checks prop declaration + composition utility (`filter(Boolean).join()`, `clsx`, `cn`, `classnames`) |
| `variant-union` | `variant`/`size` props must use string literal unions, not bare `string` | Regex detecting `variant?: string` vs `variant?: 'a' \| 'b'` |
| `token-exists` | All referenced CSS tokens must exist in `tokens.css` | Cross-references `var(--*)` usage against `--*:` definitions |

### Warning Rules (4) — Advisory recommendations

| Rule ID | Purpose | Detection Method |
|---------|---------|-----------------|
| `three-file` | Component should have `.tsx`, `.module.css`, `.figma.tsx` | File existence checks |
| `barrel-export` | Component should be exported from parent `index.ts` | Regex pattern match in barrel content |
| `design-tokens` | CSS should use design token variables (`var(--*)`) | Checks for `var(--` patterns; detects hardcoded colors/sizes |
| `html-attrs` | Props should extend `HTMLAttributes` or `React.ComponentProps` | Regex for inheritance declarations |
| `dsl-compatible-layout` | CSS should use `display: flex` or `display: grid` | Presence check for flex/grid patterns |

**Evidence**: `src/rules/index.ts:1-35`, all rule files

## Pattern Matching Logic
**Confidence**: 0.95 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

All rules use **regex-based pattern matching** rather than AST parsing:

- **CSS-Modules**: Detects `import styles from './Component.module.css'` format
- **No-Inline-Style**: Line-by-line scan for `style={{` with line number reporting
- **Classname-Prop**: Two-step check — presence of `className` in type definition + composition pattern
- **Variant-Union**: Detects `variant?: string` (bare) vs `variant?: 'primary' | 'secondary'` (literal union)
- **Token-Exists**: Builds a `Set` of defined tokens from `tokens.css` (`--name:` pattern), then cross-references `var(--name)` usage in CSS
- **Design-Tokens**: Uses negative lookbehind to detect hardcoded colors not wrapped in `var()`: `/(?<!var\()#[0-9a-fA-F]{3,8}\b/`
- **HTML-Attrs**: Matches `extends HTMLAttributes`, `ButtonHTMLAttributes`, `React.ComponentPropsWithoutRef`, etc.

**Design rationale**: Regex over AST avoids TypeScript/Babel parser dependencies, keeping the validator lightweight with zero external dependencies. Trade-off: less semantic precision, potential false positives in string literals and comments.

**Evidence**: All rule files in `src/rules/`

## Batch Validation
**Confidence**: 0.97 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

`validateAll()` (validator.ts lines 100-126):

1. Reads directory entries from `baseDir`
2. Filters directories where `{dirName}/{dirName}.tsx` exists (component detection)
3. Validates each matching directory via `validateComponent()`
4. Returns `Map<string, ValidationResult>` keyed by component name
5. Returns empty map if `baseDir` doesn't exist (graceful degradation)
6. Validation is sequential (not parallelized)

## Error Handling & Resilience
**Confidence**: 0.93 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

### File I/O

- Uses `existsSync()` before `readFileSync()` — defensive checks
- Missing files result in `undefined` context fields (not thrown)
- Rules check `if (!context.field)` before processing (graceful degradation)

### Rule Execution

- Rules return empty arrays on missing prerequisites (never throw)
- No try-catch around individual `rule.validate()` calls — a rule crash would crash validation
- Rules are stateless; no cleanup required

### CLI Error Handling

- Try-catch wrapper around all CLI commands
- Runtime errors → exit code 2 with human-readable messages

## Severity Model & Strictness
**Confidence**: 0.98 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

### Two Severity Levels

- **error** (6 rules): Violations that prevent valid component operation. Affects `result.valid`.
- **warning** (4 rules): Best-practice recommendations. Does not affect `result.valid` by default.

### Strictness

- `result.valid` is determined by **errors only** (warnings don't affect validity)
- CLI `--strict` flag elevates warnings to exit code 1

This dual-level system supports:
- Quick iteration during development (ignore warnings)
- Strict CI/CD enforcement (`--strict` flag)
- Clear signal of required vs. recommended patterns

**Evidence**: Rule definitions (severity fields), CLI `src/cli.ts:531`

## Configuration & Extensibility
**Confidence**: 0.94 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

### User Configuration

- Rule whitelist: `options.rules` — run only specified rules
- Rule blacklist: `options.skipRules` — exclude specified rules
- Token path: `options.tokensPath` — override auto-detection
- CLI flags: `--format`, `--strict` — output control

### Adding a New Rule

1. Create `src/rules/{rule-id}.ts` implementing `ValidationRule` interface
2. Import and add to `allRules` array in `src/rules/index.ts`
3. Write tests in `rules.test.ts`
4. No changes needed to core validator logic

### Limitations

- Rule severity is hardcoded — cannot be overridden at validation time
- No per-rule configuration options
- No runtime plugin system — requires code changes
- No config file support (`.validatorrc` or similar)

**Evidence**: `src/types.ts:16-23`, `src/validator.ts:61-73`, `src/rules/index.ts`

## Performance Characteristics
**Confidence**: 0.87 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

### Single Component

- Context building: 5 synchronous file reads (worst case)
- Rule execution: Sequential async loop over selected rules
- Each rule: linear scan of file content (regex matching)
- Expected runtime: <100ms per component for typical files (<10KB)

### Batch Mode

- Complexity: O(components × rules × file_size)
- Sequential — not parallelized
- No caching between components

### Memory

- Context loads entire file contents into memory
- Token-exists rule stores token names in a `Set` for O(1) lookup
- Batch results grow linearly with component count

### Test Performance

29 tests complete in ~29ms (dominated by fixture setup/teardown, not validation).

## Known Limitations
**Confidence**: 0.92 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

1. **No TypeScript AST parsing**: Rules examine raw strings via regex, not parsed types. Can produce false positives in string literals and comments.
2. **Regex pattern fragility**: Code formatted differently than expected may not be caught (e.g., multi-line type definitions, non-standard import syntax).
3. **Line number precision**: Only available for rules with line-by-line scanning; multi-line patterns report first line or no line.
4. **No cross-rule dependencies**: Rules execute independently; cannot express "run B only if A passes."
5. **Component detection pattern**: `validateAll()` only discovers directories containing `{dirName}/{dirName}.tsx` — not configurable.
6. **No config file support**: Validation options must be passed via CLI flags or API calls.
7. **Sequential execution**: Rules execute sequentially; no parallelization for batch operations.
8. **No rule crash isolation**: A rule that throws will crash the entire validation (no try-catch around individual rules).
9. **Single font metadata entry**: `classname-prop` pattern `/className[\s?]*[:\s]/` may false-match similar names like `classNameX`.
10. **Design-tokens lookbehind**: Uses `/(?<!var\()#[0-9a-fA-F]{3,8}\b/` which may not be supported in all JavaScript environments.
