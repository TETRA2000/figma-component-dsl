## Overview
**Confidence**: 0.97 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

The `@figma-dsl/comparator` package is a visual regression testing utility that performs pixel-level image comparison using the pixelmatch library. It compares PNG images with configurable sensitivity thresholds and produces similarity metrics, pass/fail determinations, and diff visualization images. The package is a leaf dependency in the figma-dsl monorepo, consumed exclusively by `@figma-dsl/cli` for render verification, batch comparison, and calibration workflows.

## Public API
**Confidence**: 0.98 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

### `compare(imageA: Buffer, imageB: Buffer, options?: Partial<CompareOptions>): CompareResult`

In-memory comparison of two PNG buffers.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `imageA` | `Buffer` | Yes | First PNG image buffer |
| `imageB` | `Buffer` | Yes | Second PNG image buffer |
| `options` | `Partial<CompareOptions>` | No | Override default thresholds and settings |

**Behavior**: Decodes both buffers → checks dimensions → pads if mismatched → runs pixelmatch → calculates similarity → returns result with diff PNG buffer.

### `compareFiles(pathA: string, pathB: string, diffOutputPath?: string, options?: Partial<CompareOptions>): CompareResult`

File-based wrapper around `compare()`.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `pathA` | `string` | Yes | Path to first PNG file |
| `pathB` | `string` | Yes | Path to second PNG file |
| `diffOutputPath` | `string` | No | Path to write diff visualization PNG |
| `options` | `Partial<CompareOptions>` | No | Override default thresholds and settings |

**Behavior**: Reads files via `readFileSync()` → calls `compare()` → optionally writes diff PNG to disk (with `mkdirSync({ recursive: true })`).

### `CompareResult` Interface

```typescript
interface CompareResult {
  similarity: number;         // 0–100 percentage
  mismatchedPixels: number;   // Count of differing pixels
  totalPixels: number;        // width × height of padded image
  dimensionMatch: boolean;    // Whether original dimensions matched
  pass: boolean;              // similarity >= failThreshold
  diffPngBuffer?: Buffer;     // PNG-encoded difference visualization
}
```

Note: `diffPngBuffer` is marked optional in the interface but is always present in practice.

## Configuration & Thresholds
**Confidence**: 0.96 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

### `CompareOptions` Interface

```typescript
interface CompareOptions {
  threshold: number;                            // Default: 0.1
  failThreshold: number;                        // Default: 95
  diffColor: { r: number; g: number; b: number }; // Default: { r: 255, g: 0, b: 0 }
  antiAliasing: boolean;                        // Default: true
}
```

| Option | Default | Description |
|--------|---------|-------------|
| `threshold` | 0.1 | Pixelmatch sensitivity [0–1); lower = more sensitive to pixel differences |
| `failThreshold` | 95 | Minimum similarity % (0–100) for `pass: true` |
| `diffColor` | Red | RGB color for highlighting mismatched pixels in diff PNG |
| `antiAliasing` | true | When true, pixelmatch ignores anti-aliased pixels (more lenient) |

**Important**: The `antiAliasing` option is semantically inverted when passed to pixelmatch — `antiAliasing: true` sets `includeAA: false` in pixelmatch, which makes the comparison *more lenient*.

Options are merged with defaults via spread operator: `{ ...DEFAULT_OPTIONS, ...options }`. Missing fields silently fall back to defaults.

**Evidence**: `src/comparator.ts:6-11,22-27,38,65`

## Image Processing Pipeline
**Confidence**: 0.96 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

The `compare()` function processes images through this pipeline:

```
PNG Buffer A ─→ PNG.sync.read() ─→ PNG Object (width, height, data)
PNG Buffer B ─→ PNG.sync.read() ─→ PNG Object (width, height, data)
                                         │
                                    Dimension Check
                                   ┌─── match ───┐
                                   │              │
                              Use as-is      padImage() to max(W,H)
                                   │              │
                                   └──── merge ───┘
                                         │
                                    pixelmatch()
                                         │
                                   ┌─────┴─────┐
                              mismatchedPixels  diff buffer
                                   │              │
                              Calculate %    PNG.sync.write()
                                   │              │
                              CompareResult ← diffPngBuffer
```

**Similarity formula**: `(totalPixels - mismatchedPixels) / totalPixels × 100`

Edge case: Returns 100% similarity when `totalPixels === 0` (prevents division by zero).

**Evidence**: `src/comparator.ts:29-82`

## Dimension Mismatch & Padding
**Confidence**: 0.94 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

When images have different dimensions, the smaller image is padded to match the larger:

1. New canvas allocated at `max(widthA, widthB) × max(heightA, heightB)`
2. Filled entirely with opaque white (RGBA 255,255,255,255)
3. Original image pixels copied to top-left corner
4. Padded image used for comparison

**Design implications**:
- Dimension mismatch is **not** treated as a failure — comparison proceeds with padding
- The `dimensionMatch` flag in the result indicates whether padding was applied
- White padding affects similarity scores (unmatched white regions count as differences if the other image has non-white pixels there)
- Padding color (white) is hardcoded, not configurable
- `padImage()` returns the original unchanged if dimensions already match (optimization)

**Evidence**: `src/comparator.ts:43-52,102-127`

## Error Handling & Failure Modes
**Confidence**: 0.90 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

The comparator uses a minimal error handling strategy — errors propagate to the caller:

| Scenario | Error Source | Behavior |
|----------|-------------|----------|
| Invalid PNG buffer | `pngjs` throws | Propagates to caller |
| File not found | `readFileSync` throws ENOENT | Propagates to caller |
| Permission denied (write) | `writeFileSync` throws EACCES | Propagates to caller |
| Disk full | `writeFileSync` throws ENOSPC | Propagates to caller |
| Invalid option values | None | Passed directly to pixelmatch; undefined behavior |

No try-catch wrapping in any function. The `decodePng()` helper delegates directly to `PNG.sync.read()` without error handling. Callers (the CLI) are responsible for catching and handling errors.

**Evidence**: `src/comparator.ts:29-31,84-100`

## Integration Patterns
**Confidence**: 0.94 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

### CLI Integration

The comparator is consumed by three CLI commands:

1. **`compare`**: Single image pair comparison via `compareFiles()`
2. **`batch-compare`**: Directory-level comparison, iterating `compareFiles()` per component
3. **`pipeline`**: Final stage comparing DSL render vs captured React component

### Data Flow in Monorepo

```
DSL Definition (.dsl.ts)
    ↓ compile()
Compiled Nodes
    ↓ render()
PNG Buffer (RenderResult.pngBuffer)
    ↓ compare()
CompareResult { similarity, pass, diffPngBuffer }
```

The comparator is a **terminal module** in the visual regression pipeline — it consumes PNG buffers and does not feed data to other packages.

### Calibration Reporter

`calibration-reporter.ts` in the CLI uses `compareFiles()` with custom thresholds and diff output directories. Results are aggregated into JSON reports with per-category breakdown.

**Evidence**: CLI `src/cli.ts:6`, `src/calibration-reporter.ts`

## Performance Characteristics
**Confidence**: 0.90 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

### Complexity

- **Time**: O(width × height) for pixel comparison (linear in image area)
- **Space**: ~3× image size in memory (two decoded PNGs + diff buffer)

### Memory Estimates

| Image Size | Per-Image Memory | Total Comparison |
|------------|-----------------|-----------------|
| 50×50 | ~10 KB | ~30 KB |
| 1920×1080 | ~8.3 MB | ~25 MB |
| 4096×2160 | ~35 MB | ~105 MB |

### Important Characteristics

- All operations are **synchronous** (`PNG.sync.read/write`, `readFileSync/writeFileSync`) — blocks the event loop
- Diff PNG buffer is **always generated**, even when `diffOutputPath` is not provided
- No image size limits enforced — large images will consume proportional memory
- No parallelism support — each comparison runs sequentially

**Evidence**: `src/comparator.ts:29-82,102-127`

## Known Limitations
**Confidence**: 0.92 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

1. **Synchronous only**: All operations block the event loop — not suitable for high-throughput server use.
2. **Always generates diff**: Diff PNG buffer allocated even when not needed, wasting memory.
3. **Hardcoded padding color**: White background padding is not configurable.
4. **No input validation**: Invalid buffers, negative thresholds, or extremely large images are not checked.
5. **No size limits**: Arbitrarily large images accepted with proportional memory consumption.
6. **Interface mismatch**: `diffPngBuffer` marked optional in `CompareResult` but always present in implementation.
7. **Inverted semantics**: `antiAliasing: true` actually makes comparison *less* strict (non-obvious behavior).
