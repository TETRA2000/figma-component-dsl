## Overview
**Confidence**: 0.97 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

The `@figma-dsl/core` package is the foundational module of the Figma Component DSL system. It provides a fully-typed AST (Abstract Syntax Tree) layer that models Figma design elements programmatically. The package exports factory functions for creating design nodes (`frame`, `text`, `rectangle`, `ellipse`, `group`, `component`, `componentSet`, `instance`), utility functions for color handling (`hex`, `solid`, `gradient`, `defineTokens`, `token`), and configuration helpers for auto-layout (`horizontal`, `vertical`).

All modules are stateless — no runtime state is maintained. The design enables developers to compose complex component hierarchies declaratively using TypeScript.

## Color & Fill System
**Confidence**: 0.97 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

### Functions

#### `hex(value: string): RgbaColor`
Parses 6-digit hex strings (`"#ff0000"` or `"ff0000"`). Returns `{r, g, b, a}` with values normalized to 0–1. Alpha is always 1.0. Throws on invalid input with descriptive error message.

#### `solid(hexValue: string, opacity = 1): SolidFill`
Convenience wrapper creating `{type: 'SOLID', color, opacity, visible: true}`.

#### `gradient(stops: {hex: string, position: number}[], angle = 0): GradientFill`
Creates `GRADIENT_LINEAR` fill. Stops use 0–1 positions. Angle in degrees (0° = left-to-right). Transform matrix computed via rotation around center (0.5, 0.5) in UV space.

#### `defineTokens(tokens: Record<string, string>): ColorTokenMap`
Eagerly parses all hex values at definition time. Returns opaque `ColorTokenMap` with `readonly _tokens` field.

#### `token(map: ColorTokenMap, name: string): SolidFill`
Looks up token by name. Throws descriptive error listing available tokens if not found.

### Fill Types

```typescript
type Fill = SolidFill | GradientFill;  // Discriminated union via 'type' field

interface SolidFill {
  type: 'SOLID'; color: RgbaColor; opacity: number; visible: boolean;
}

interface GradientFill {
  type: 'GRADIENT_LINEAR'; gradientStops: GradientStop[];
  gradientTransform: number[][]; opacity: number; visible: boolean;
}
```

## Auto-Layout & Child Layout Props
**Confidence**: 0.96 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

### Layout Helpers

#### `horizontal(config?): AutoLayoutConfig`
Sets `direction: 'HORIZONTAL'` and merges partial config. The `Omit<AutoLayoutConfig, 'direction'>` type prevents accidental direction override.

#### `vertical(config?): AutoLayoutConfig`
Sets `direction: 'VERTICAL'` with same merge behavior.

## Component Properties & Variants
**Confidence**: 0.95 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

### ComponentProperty

```typescript
interface ComponentProperty {
  name: string;
  type: 'TEXT' | 'BOOLEAN' | 'INSTANCE_SWAP';
  defaultValue: string | boolean;
  preferredValues?: string[];  // INSTANCE_SWAP only
}
```

- **TEXT**: String default for swappable text values
- **BOOLEAN**: Boolean default for conditional visibility
- **INSTANCE_SWAP**: Component ref default + optional `preferredValues` array

### Variant System

- `componentSet()` accepts `variantAxes: Record<string, string[]>`
  - Key = axis name (e.g., `"Variant"`, `"Size"`)
  - Value = option names (e.g., `['Primary', 'Secondary']`)
- Variant children use `"Key=Value, Key=Value"` naming convention
- Example: `component('Variant=Primary, Size=Large', { ... })`

### Instance System

- `instance(componentRef, overrides?)` creates a reference to a component
- `propertyOverrides: Record<string, string | boolean>` maps property names to values
- `componentRef` must be non-empty (validated at runtime)

## Type System & AST Structure
**Confidence**: 0.96 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

### DslNode — Central AST Interface

All node types share a unified `DslNode` interface with optional type-specific fields:

**Base properties** (all types):
- `type`, `name`, `size?`, `fills?`, `strokes?`, `opacity?`, `visible?`, `children?`

**Container properties** (FRAME, COMPONENT):
- `autoLayout?`, `clipContent?`, `cornerRadius?`, `cornerRadii?`

**Child layout** (any node in auto-layout parent):
- `layoutGrow?`, `layoutSizingHorizontal?`, `layoutSizingVertical?`

**Text** (TEXT only):
- `characters`, `textStyle?`

**Component** (COMPONENT only):
- `componentProperties?`

**Component Set** (COMPONENT_SET only):
- `variantAxes?`

**Instance** (INSTANCE only):
- `componentRef`, `propertyOverrides?`

### Size Representation
Sizes use `{x: number, y: number}` (not width/height).

### NodeType Union
```typescript
type NodeType = 'FRAME' | 'TEXT' | 'RECTANGLE' | 'ELLIPSE' | 'GROUP'
  | 'COMPONENT' | 'COMPONENT_SET' | 'INSTANCE'
```

## Immutability & Defensive Copying
**Confidence**: 0.96 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

All factory functions apply defensive copying to mutable array inputs:

```typescript
fills: props.fills ? [...props.fills] : undefined
children: props.children ? [...props.children] : undefined
strokes: props.strokes ? [...props.strokes] : undefined
componentProperties: props.componentProperties ? [...props.componentProperties] : undefined
```

This prevents caller mutations from affecting created nodes. Verified by tests: pushing to the original array after construction does not affect the node.

**Non-copied**: Fill and TextStyle objects are referenced directly (not deep-cloned). Callers are expected not to mutate them after passing.

All factories are pure functions — same inputs always produce structurally identical outputs, with no side effects or global state.

**Evidence**: `src/nodes.ts:21,26,78,110,122,132,147`, `src/nodes.test.ts:38-44`
