## Overview
**Confidence**: 0.96 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

The `@figma-dsl/cli` package is the command-line interface that orchestrates the entire Figma Component DSL workflow. It provides 12 commands spanning core DSL operations (compile, render, capture, compare), batch processing, calibration workflows, validation, and Figma API integration. The CLI connects 7 upstream packages (`compiler`, `renderer`, `capturer`, `comparator`, `exporter`, `validator`, `dsl-core`) into a cohesive pipeline, serving as the primary entry point for developers and CI/CD systems.

## Command Reference
**Confidence**: 0.95 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

### Core Workflow Commands

#### `compile` — DSL to JSON
```bash
figma-dsl compile <file.dsl.ts> [-o output.json]
```
Loads a `.dsl.ts` module, compiles with layout measurements, outputs JSON. Errors are reported with `[nodeType] nodePath: message` format.

#### `render` — DSL/JSON to PNG
```bash
figma-dsl render <file.dsl.ts|compiled.json> -o output.png [-s scale] [-b background] [--debug-layout]
```
Accepts either raw DSL (compiled on-the-fly) or precompiled JSON. Scale is a float multiplier (default 1.0).

The `--debug-layout` flag overlays layout debug information on the rendered PNG:
- Semi-transparent colored rectangles for padding areas
- Dashed red borders around every frame
- Frame names and computed sizes in small text

#### `capture` — URL to PNG
```bash
figma-dsl capture <url> -o output.png [-v WxH] [--selector css]
```
Screenshots a URL using headless Chromium via `@figma-dsl/capturer`. Default viewport: 1280x720.

#### `compare` — Visual diff
```bash
figma-dsl compare <imageA.png> <imageB.png> [-t threshold] [-d diff.png]
```
Compares two PNGs and reports similarity percentage. Default threshold: 95%. Exit code 1 if below threshold.

#### `pipeline` — Full 4-stage automation
```bash
figma-dsl pipeline <file.dsl.ts> -u <react-url> [-o dir] [-t threshold] [-v WxH]
```
Chains: DSL compile → render PNG → capture React URL → compare. Exit code 0/1 based on comparison.

#### `export` — Generate Figma plugin input
```bash
figma-dsl export <file.dsl.ts> -o plugin-input.json [-p "Page Name"]
```
Compiles DSL and transforms to Figma plugin-compatible JSON format.

### Batch & Calibration Commands

#### `batch` — Bulk compile/render/export
```bash
figma-dsl batch <dir|glob> -o <output-dir> [--include <path>...] [-p page] [-s scale]
```
Processes multiple `.dsl.ts` files. Outputs: `dsl/` (PNGs), `plugin-input.json` (merged), `batch-manifest.json`.

#### `batch-compare` — Compare PNG directories
```bash
figma-dsl batch-compare <dsl-dir> <figma-dir> [-o report.json] [-t threshold] [--diff-dir dir]
```
Pairs PNGs by filename between directories. Generates JSON report with category breakdown.

#### `capture-figma` — Download from Figma API
```bash
figma-dsl capture-figma -o <dir> --file-key <key> --token <token> --node-id-map <path> [-s scale]
```
Fetches component PNG renders from Figma REST API with retry logic.

#### `calibrate` — Full calibration pipeline
```bash
figma-dsl calibrate [-o dir] [--property cat] [--skip-generate] [--capture-mode api] \
  [--file-key key] [--token token] [--node-id-map path] [-t threshold]
```
4-stage pipeline: generate test suite → batch process → capture Figma → compare.

#### `generate-test-suite` — Create test files
```bash
figma-dsl generate-test-suite -o <output-dir> [--property <category>...]
```
Generates `.dsl.ts` test variants organized by property category.

### Validation Command

#### `validate` — Component DSL compatibility check
```bash
figma-dsl validate <path> [--tokens path] [--rules list] [--skip list] [--format text|json] [--strict]
```
Validates components for DSL compatibility. Supports single component or directory-wide validation. Exit code 1 if errors found (or warnings in strict mode).

## Module Loading & Dynamic Imports
**Confidence**: 0.97 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

DSL files are dynamically imported at runtime using Node's ES module loader:

```typescript
async function loadDslModule(dslPath: string): Promise<DslNode> {
  const absolutePath = resolve(dslPath);
  const fileUrl = pathToFileURL(absolutePath).href;
  const mod = await import(fileUrl) as { default?: DslNode };
  if (!mod.default) {
    throw new Error(`DSL module must export a default DslNode: ${dslPath}`);
  }
  return mod.default;
}
```

**Requirements**:
- File must be a valid ES module (TypeScript `.dsl.ts` or JavaScript)
- Must have a `default` export of type `DslNode`
- Path is resolved relative to `process.cwd()` and converted to a `file://` URL

This pattern appears in both `cli.ts` and `batch-processor.ts`.

**Evidence**: `src/cli.ts:40-48`, `src/batch-processor.ts:49-57`

## Font Initialization & Service Setup
**Confidence**: 0.96 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

The CLI initializes two shared services before most commands:

```typescript
async function initServices(): Promise<void> {
  const fontDir = findFontDir();
  await textMeasurer.initialize(fontDir);
  await initializeRenderer(fontDir);
}
```

**Font directory discovery** searches three candidate paths in order:
1. `../../dsl-core/fonts` (relative to compiled `bin.js`)
2. `packages/dsl-core/fonts` (relative to `cwd`)
3. `node_modules/@figma-dsl/core/fonts`

Returns the first existing path. If none exists, returns the first candidate anyway — this is a potential silent failure point identified by all agents.

## Compilation & Layout Workflow
**Confidence**: 0.97 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

Commands that compile DSL follow this pattern:
1. Load DSL module via `loadDslModule()`
2. Compile with `compileWithLayout(dslNode, textMeasurer)` — produces `CompileResult { root, errors }`
3. Report compilation errors to stderr with `[nodeType] nodePath: message` format
4. Proceed with compiled output or fail based on error severity

Compilation errors are non-fatal for the output (errors included in JSON), but logged for developer visibility. The `pipeline` command returns exit code 2 if compilation errors exist.

## Batch Processing System
**Confidence**: 0.96 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

The `processBatch()` function orchestrates bulk operations:

1. **File Discovery**: `discoverDslFiles()` finds `.dsl.ts` files via glob patterns
2. **Per-file processing** (sequential):
   - Load DSL module → compile → render PNG → generate plugin input
3. **Artifact aggregation**:
   - Merge all plugin nodes into single `plugin-input.json`
   - Write `batch-manifest.json` with component metadata

**Output structure**:
```
<outputDir>/
├── dsl/
│   ├── component-name.png
│   └── ...
├── plugin-input.json  (merged)
└── batch-manifest.json
```

**Error isolation**: If one component fails, processing continues. Failed components are marked `status: 'error'` in the manifest.

**Batch Manifest** includes: timestamp, input pattern, output directory, per-component results (name, paths, status, error, dimensions), and summary counts.

**Evidence**: `src/batch-processor.ts:59-149`, `src/glob-util.ts:9-58`

## Figma API Integration
**Confidence**: 0.95 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

### Retry Strategy

`fetchWithRetry()` implements exponential backoff:
- **403 Forbidden**: Throws immediately (authentication error)
- **429 Rate Limited**: Waits `2^(attempt+1)` seconds, retries up to 3 times
- **Other errors**: Waits and retries; throws after `maxRetries`

### Batch Image Capture

`captureFigmaImages()` workflow:
1. Single batch request to `/images/{fileKey}` with all node IDs
2. Individual download of each returned image URL
3. PNG dimensions extracted via `pngjs`
4. Missing/failed images collected in `missingComponents[]`

**Requirements**:
- Figma personal access token
- File key (from Figma URL)
- Node ID map: JSON file mapping component names to Figma node IDs (`{ "button": "123:456" }`)

**Evidence**: `src/figma-capturer.ts:22-132`

- **Single component**: Detects component directory (contains `{name}.tsx`), calls `validateComponent()`
- **Bulk validation**: Validates all components in a directory, calls `validateAll()`

**Options**: `--tokens` (path to token file), `--rules`/`--skip` (comma-separated rule lists), `--strict` (treat warnings as failures)

**Exit codes**: 0 if all pass (and no warnings in strict mode), 1 if errors or strict-mode warnings.

## Error Handling & Exit Codes
**Confidence**: 0.97 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

### Exit Code Semantics

| Code | Meaning | Used By |
|------|---------|---------|
| 0 | Success | All commands |
| 1 | Logical failure (comparison/validation below threshold) | compare, pipeline, batch-compare, calibrate, validate |
| 2 | Runtime/usage error (missing args, file not found) | All commands |

### Error Patterns

1. **Command-level**: All commands wrap operations in try-catch, log to stderr, return code 2
2. **Per-item isolation**: Batch operations catch per-item errors and continue processing
3. **Retry with state**: Figma API preserves last error and throws after exhausting retries
4. **Graceful degradation**: File discovery swallows read errors and returns empty results

### Errors that propagate
- Missing DSL `default` export (includes file path in message)
- Compilation errors (logged with `[nodeType] nodePath: message`)
- Figma 403 authentication errors (thrown immediately)
- File system errors (permission denied, disk full)

**Evidence**: `src/cli.ts:89-93,199,265`, `src/figma-capturer.ts:22-66`, `src/batch-processor.ts:100-110`

## Known Limitations
**Confidence**: 0.93 | **Consensus**: Full | **Sources**: Architect, Developer, Analyst

1. **Font initialization**: Font directory is not validated for existence — could fail silently during text measurement if no candidate path exists.
2. **Figma API timeout**: No timeout on `fetch()` calls — could hang indefinitely on network issues.
3. **Glob simplicity**: Custom glob doesn't support `{a,b}` patterns or advanced features; falls back to base directory recursion.
4. **Partial batch success**: Exit code 0 if at least one component succeeds, even if others fail — may mask errors.
5. **No output validation**: Output directories are not checked for writability before processing starts.
6. **TypeScript only**: DSL files must be valid TypeScript ES modules — no syntax validation before import.
7. **Sequential processing**: Batch operations process files sequentially — no parallelism for multi-core utilization.
